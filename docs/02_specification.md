# Спецификация требований telegram-join-manager

## 1. Общие положения

### 1.1. Назначение спецификации
Настоящая спецификация описывает функциональные и нефункциональные требования к системе `telegram-join-manager`.  
Спецификация используется для реализации, тестирования и сопровождения системы.

### 1.2. Объекты и термины
- **Пользователь:** участник Telegram, подающий join‑запрос на вход в приватный чат/канал.
- **ChatJoinRequest:** событие от Telegram API, сигнализирующее о запросе пользователя на вступление.
- **Бот (telegram-join-manager):** автоматический Telegram‑бот, обрабатывающий `ChatJoinRequest`.
- **Капча:** проверка, которую пользователь должен пройти для подтверждения запроса.
- **FSM (Finite State Machine):** конечный автомат, описывающий состояние пользователя (ожидание ответа, капча решена и т.п.).

---

## 2. Функциональные требования

### 2.1. Обработка запроса на вступление
- В системе должно быть возможно подключение к одному или нескольким приватным чатам/каналам, в которых включён режим `join_request`.
- Бот должен:
  - принимать `ChatJoinRequest` для заданных чатов;
  - фиксировать факт запроса в базе данных;
  - отправлять пользователю сообщение‑капчу.

### 2.2. Логика капчи
- Капча представляет собой задание (например, арифметический пример, вопрос, текст‑код и т.д.), определяемое конфигурацией.
- Пользователь должен ответить в течение заданного таймаута.
- Если ответ корректен:
  - бот подтверждает запрос (`approveChatJoinRequest`).
  - записывает в БД статус `accepted`.
- Если ответ неверный или не был прислан вовремя:
  - бот отклоняет запрос (`declineChatJoinRequest`);
  - записывает в БД статус `rejected`.

### 2.3. Хранение данных
- В БД должен храниться:
  - ID пользователя Telegram;
  - ID чата/канала;
  - дата создания запроса;
  - статус запроса (`pending`, `accepted`, `rejected`);
  - временная отметка результата.
- Схема БД должна поддерживать:
  - поиск по пользователю;
  - фильтрацию по статусу и дате.

### 2.4. Работа с администраторами и разработчиками
- В системе должны быть выделены роли:
  - *разработчик* — имеет доступ к конфигурации, логам и настройкам;
  - *администраторы* — могут управлять ботом через Telegram‑интерфейс (например, включать/выключать чаты, просматривать статистику).
- Администратор должен иметь интерфейс:
  - просмотра количества заявок за выбранную дату;
  - просмотра соотношения `accepted` / `rejected`.

---

## 3. Нефункциональные требования

### 3.1. Стабильность
- Система должна быть готова к длительной работе без перезапуска (> 7 дней).
- Обработка сбоев сети и ошибок API Telegram должна быть прозрачной: повторные попытки и логирование.

### 3.2. Производительность
- Время обработки одного `ChatJoinRequest` (до ответа пользователю) — не более 10 секунд.
- Время подтверждения/отклонения запроса — не более 5 секунд.
- Система должна корректно обрабатывать до 1000 запросов в час на один экземпляр бота.

### 3.3. Масштабируемость
- Система должна допускать:
  - использование нескольких ботов для разных чатов;
  - распределение нагрузки между экземплярами бота;
  - использование внешних БД (PostgreSQL/MySQL) вместо SQLite.

### 3.4. Безопасность
- Секретные данные (`BOT_TOKEN`, `API_ID`, `API_HASH`) не должны быть в репозитории.
- Доступ к БД и Redis должен быть ограничен и защищён.
- Входные данные (ID чатов, пользователей, ссылки) должны проверяться на корректность.

---

## 4. Требования к интерфейсу

### 4.1. Интерфейс пользователя
- Пользователь получает:
  - сообщение‑капчу после отправки join‑запроса;
  - информационное сообщение о результате (в случае реализации).

### 4.2. Интерфейс администратора
- Через Telegram‑чат с ботом администратору доступны:
  - команды `/stats`, `/chats`, `/pause`, `/resume`;
  - простой текстовый вывод статистики и уведомлений.
- Команды должны быть задокументированы в `admin_guide.md`.

---

## 5. Требования к данным

- Данные о запросах:
  - `user_id`, `username` (если есть);
  - `chat_id`, `chat_title`;
  - `request_date`, `status` (`pending`, `accepted`, `rejected`);
  - `answer_time` (если ответ предоставлен).
- Данные должны быть:
  - сохранены в реляционной БД;
  - доступны для экспорта/резервного копирования.
