# Тестирование telegram-join-manager

## Unit‑тесты

- Проверка валидации `.env`‑конфигурации.
- Проверка корректной работы FSM‑состояний (`waiting_captcha`, `accepted`, `rejected`).
- Тестирование модели БД (CRUD‑операций для `applications`).

## Интеграционные тесты

- Проверка соединения с Redis и SQLite/PostgreSQL.
- Проверка отправки/получения сообщений через `aiogram` mock‑клиент.
- Проверка обработки `ChatJoinRequest`‑событий без реального Telegram API (через тестовый фреймворк).

## Нагрузочное тестирование

### Сценарий
1. Создать группу с режимом `join_request`.
2. Запустить бота.
3. Эмулировать N join‑запросов в час (через тестовый клиент).
4. Измерить:
   - время ответа капчи;
   - количество ошибок;
   - пикову нагрузку на CPU/Redis/БД.

### Цели
- Подтвердить возможность работать при 100–1000 requests per hour.
- Определить порог нагрузки, после которого система начинает падать в ошибки.

Рекомендуется использовать:
- `pytest` + `pytest-asyncio`.
- `Locust` или аналогичный инструмент для нагрузочных тестов.
